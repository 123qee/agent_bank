# 养老规划 Agent 项目计划

## 目标

在 `agent_bank` 下实现一个可提交到任务 2 的 Python Agent 项目。最终提交目录根目录必须包含 `run.py`，支持：

```bash
python3 run.py "客户V500001现在年龄多大？"
```

标准输出只打印最终答案，例如 `22岁`，不输出 JSON、日志、解释或调试信息。同时保留 `run(inf: str) -> str`，方便评测程序直接导入调用。

## 提交文件规划

- `run.py`：主入口，包含命令行参数读取、`run(inf)`、问题路由、数据库查询、财务测算和答案格式化。
- `README.md`：说明运行方式、环境变量、提交注意事项和示例问题。
- `requirements.txt`：默认不创建；如后续确实使用 `requests` 调 one-api，再只声明 `requests`。

为降低判题时加载失败风险，第一版实现优先把核心逻辑集中在 `run.py`，不依赖提交目录外的文件。

## 数据库约束

训练集和评测集均通过 MySQL 查询，使用 `pymysql`，不再做 CSV、SQLite 或 Parquet 适配。连接参数和表名必须全部从环境变量读取，并提供本地开发默认值：

```python
DB_HOST = os.getenv("TASK2_DB_HOST", "172.16.48.27")
DB_PORT = int(os.getenv("TASK2_DB_PORT", "3306"))
DB_USER = os.getenv("TASK2_DB_USER", "test_user")
DB_PASSWORD = os.getenv("TASK2_DB_PASSWORD", "R6#pV9@kT3!xM2$q")
DB_NAME = os.getenv("TASK2_DB_NAME", "cmb_contest")
BASE_TABLE = os.getenv("TASK2_BASE_TABLE", "train_base_table")
ACTION_TABLE = os.getenv("TASK2_ACTION_TABLE", "train_action_table")
```

SQL 中只允许通过 `BASE_TABLE`、`ACTION_TABLE` 拼接表名，不硬编码训练表或测试表。查询参数使用 `%s` 占位，避免把客户 ID 直接拼进 SQL。

## 字段映射

客户基础信息表字段按题面最新说明使用：

- `User_ID`：客户标识
- `Age`：年龄
- `Gender`：性别
- `Rsk_Cd`：风险评级
- `Net_Asset`：净资产
- `Monthly_Income`：月收入
- `Monthly_Expend`：月支出
- `Pension`：退休金，每月预计值
- `Enterprise_Ann`：企业年金，一次性提取预计值

客户行为信息表字段使用：

- `user_id`
- `action_typ`
- `prod_sub_typ`
- `prod_typ`
- `rsk_lvl`
- `acs_tm`

产品库映射按题面规则固化：

- `prod_sub_typ='现金'` 映射为现金理财。
- `prod_sub_typ='一般性' and prod_typ='存款'` 映射为定期存款。
- `prod_typ in ('基金', '理财') and rsk_lvl='R2'` 映射为短债类产品。
- `prod_typ in ('基金', '理财') and rsk_lvl='R3'` 映射为固收+产品。
- `prod_typ='基金' and rsk_lvl in ('R4', 'R5')` 映射为权益类产品。
- `prod_sub_typ in ('税延养老年金', '养老年金') and prod_typ='保险'` 映射为年金险。

## 核心能力

1. KYC 查询与统计

   识别客户 ID 和问题意图，回答年龄、性别、风险评级、净资产、月收入、月支出、月结余、养老金、企业年金等直接查询问题。聚合问题通过 SQL 完成，例如统计年龄大于等于某值的客户数。

2. 行为偏好分析

   行为表只按目标客户或聚合条件查询，不全量拉取。通过 SQL `CASE WHEN` 完成产品映射，再按次数排序，回答“行为最多”“最可能偏好”“浏览某类产品次数”等问题。

3. 退休测算

   使用题面固定假设：

   - 当前日期为 `2025-03-31`，假设当天生日。
   - 默认通胀率年化 `2%`，按月调整。
   - 默认投资回报率年化 `2%`，按月复利。
   - 默认预期寿命 `80` 岁。
   - 身份默认为干部。
   - 男职工原退休年龄 `60` 岁，女干部原退休年龄 `55` 岁；从 2025 年起每 4 个月延迟 1 个月，上限延迟 36 个月。

   输出覆盖：

   - 距离退休还有多久。
   - 退休时每月支出。
   - 退休时最低需要积攒金额。
   - 退休时预计可积攒金额。
   - 是否能达成养老目标及调整建议。

4. 情景测算

   支持识别常见假设表达，例如“如果 10 年后通胀率提升到 3% 并维持不变”“预期寿命延长到 90 岁”。假设类问题只影响当前回答，不写入跨问题状态。

5. 资产配置建议

   产品收益率按题面示例采用：

   - 现金理财：`1.5%`
   - 定期存款：`2.0%`
   - 短债类产品：`2.4%`
   - 固收+产品：`4.25%`
   - 权益类产品：`6.0%`
   - 年金险：`2.5%`

   根据客户 `Rsk_Cd` 过滤可投产品。收益最大化问题选择风险等级允许范围内收益率最高产品 100%；最小化风险波动问题先计算覆盖养老金缺口所需主力产品比例，剩余部分配置现金理财和年金险。

6. 未来购买预测

   优先使用客户历史购买行为对应的产品库类别；如果购买记录不足，则使用浏览、收藏、持仓等行为频次综合排序。保持确定性、低耗时，不依赖外部模型。

7. 建议书生成

   生成 7 个章节：

   - 基本情况
   - 基本假设
   - 养老目标
   - 退休后财富需求测算
   - 产品偏好
   - 资产配置方式与具体方案
   - 其他建议

   建议书默认基于当前问题、数据库和可计算结果生成。由于正式并行评测要求每次调用独立运行，不在本地维护跨进程会话状态；如题目在单进程内导入并连续调用 `run(inf)`，可使用进程内轻量记忆，但命令行并行调用不能依赖该能力。

## 大模型调用策略

默认不调用大模型，优先用 SQL、规则和公式回答，以降低耗时和 token 消耗。

仅当遇到开放式表述且规则答案不足时，才可选调用 one-api。调用时必须：

- `ONE_API_URL = os.getenv("ONE_API_URL", "https://one-api-other.nowcoder.com/v1/chat/completions")`
- `ONE_API_KEY` 从配置读取或按题面要求填入，不把平台代理地址硬编码到业务逻辑外。
- 模型使用 `qwen3.6-flash` 或 `doubao-seed-2-0-lite-260428-cmb`。
- 请求和异常信息不输出到标准输出。

第一版可以不依赖 one-api，因此不需要 `requests` 和 `requirements.txt`。

## 并行评测与运行安全

- 每次命令行执行只读取环境变量、连接数据库、计算当前问题并打印答案。
- 不修改提交目录源码、配置文件或共享数据文件。
- 不写固定文件名的缓存、日志或中间结果。
- 不依赖“第几次调用”“上一题结果”等跨进程状态。
- 数据库连接每次按需创建并在 `finally` 中关闭。
- 捕获异常时也只返回简短答案或兜底文本，不能打印 traceback 到标准输出。

## 验证重点

- `python3 run.py "客户V500001现在年龄多大？"` 输出类似 `22岁`。
- 标准输出没有日志、JSON、SQL、解释性调试文本。
- 表名随 `TASK2_BASE_TABLE`、`TASK2_ACTION_TABLE` 切换。
- 直接查询与聚合查询都通过 SQL 完成。
- 多进程并行运行不会互相覆盖文件或依赖共享状态。
